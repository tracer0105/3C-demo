# 04 开发文档

本章面向“已经准备开始写代码”的读者。  
它不是某一家厂商 SDK 的手册，而是一套尽量通用的 PC-based 运动控制开发方法。本文默认以 `EtherCAT 总线 + CiA 402 伺服驱动器 + 远程 I/O` 为主线来说明配置、回零和故障恢复。

## 1. 开发目标

一套能长期维护的 PCBase 软件，至少应该做到：

- 可以统一管理不同厂商或不同型号的轴
- 可以做回零、定位、点动、插补、联锁、报警处理
- 可以清晰记录日志、报警和状态
- 可以把业务流程和底层运动控制解耦

如果一开始就把按钮点击事件直接绑到厂商 SDK 函数上，项目很快就会失控。

## 2. 推荐软件分层

建议采用下面这类结构：

| 层级 | 职责 | 示例 |
| --- | --- | --- |
| UI / HMI 层 | 参数录入、状态显示、权限与操作界面 | WinForms、WPF、Qt、WebView |
| Application 层 | 流程编排、工站节拍、工艺逻辑 | Auto、Manual、Recipe、Vision |
| Motion 抽象层 | 统一封装轴、组轴、插补、回零、急停、状态查询 | `IMotionController`、`IAxis` |
| Driver Adapter 层 | 对接各家 SDK、DLL、驱动器对象字典 | ADLINK/Advantech/Beckhoff/ZMotion 适配器 |
| Infrastructure 层 | 日志、配置、数据库、网络通信、报警存储 | Serilog、SQLite、TCP、OPC UA |

核心原则只有一条：

`业务代码不要直接依赖厂商 SDK。`

## 3. 推荐目录结构

下面是一个适合 C# 或 C++ 项目的结构示例：

```text
PCBase/
  src/
    PCBase.App/              # UI 与应用入口
    PCBase.Application/      # 工艺流程、配方、任务编排
    PCBase.Motion.Abstractions/

    PCBase.Motion.Adapters/  # 不同厂商适配器

    PCBase.Logging/
    PCBase.Persistence/

    io.json
  docs/
  logs/
```

即便你现在还没有真正代码仓库，这个结构也值得在文档阶段先统一。

## 4. 轴抽象应该长什么样

不要在业务代码里到处写某家 SDK 的 `Move`, `Stop`, `GetCommand`, `GetActual`。  
建议先定义统一接口。

### 4.1 轴接口示例

```csharp
public interface IAxis
{
    string Name { get; }
    AxisState State { get; }
    double CommandPosition { get; }
    double ActualPosition { get; }

    Task ResetAlarmAsync(CancellationToken ct);
    Task ServoOnAsync(CancellationToken ct);
    Task ServoOffAsync(CancellationToken ct);
    Task HomeAsync(CancellationToken ct);
    Task MoveAbsoluteAsync(double position, MotionProfile profile, CancellationToken ct);
    Task MoveRelativeAsync(double distance, MotionProfile profile, CancellationToken ct);
    Task JogAsync(JogDirection direction, MotionProfile profile, CancellationToken ct);
    Task StopAsync(StopMode mode, CancellationToken ct);
}
```

这层接口的意义是：

- 换卡或换 SDK 时，上层逻辑不至于全部重写
- 更方便做模拟器和离线测试
- 更容易把单轴、组轴、龙门、虚拟轴统一建模

### 4.2 轴配置示例

```json
{
  "name": "AxisX",
  "cardType": "EtherCAT",
  "logicalId": 0,
  "homeMode": "ORG+Index",
  "positiveLimitInput": "DI_01",
  "negativeLimitInput": "DI_02",
  "homeSensorInput": "DI_03",
  "limitInputActiveLow": true,
  "homeOffset": 5.0,
  "unit": "mm",
  "pulsePerUnit": 10000,
  "softLimitMin": 0.0,
  "softLimitMax": 500.0,
  "maxVelocity": 300.0,
  "maxAcceleration": 1500.0,
  "servoOnOutput": "DO_01",
  "alarmResetOutput": "DO_02"
}
```

把这些内容配置化，比把数值硬写在代码里可靠得多。

### 4.3 EtherCAT 轴参数建议

对于 EtherCAT 伺服轴，建议把下列内容都配置化，而不是散落在界面或代码常量里：

| 参数 | 建议 | 说明 |
| --- | --- | --- |
| 轴方向 | 明确正方向与机械坐标正方向一致 | 正负方向搞反会直接影响限位和回零 |
| `positiveLimitInput` | 配成真实正极限输入 | 触发后只允许反向退回 |
| `negativeLimitInput` | 配成真实负极限输入 | 触发后只允许正向退回 |
| `limitInputActiveLow` | 优先按现场安全方案统一 | 很多现场会用常闭型限位以提高断线可检测性 |
| `softLimitMin/Max` | 设在硬限位内侧 | 先让软限位拦命令，硬限位做最后保护 |
| `homeMode` | 明确是 `ORG`、`ORG+Index` 还是驱动器 Homing | 模式不同，稳定性和精度差别很大 |
| `homeOffset` | 给最终零点留出安全偏移 | 避免停在原点开关触发沿上 |
| `maxVelocity` 与回零速度 | 分开配置 | 回零速度通常应低于正常工艺速度 |
| `Controlword/Statusword` 映射 | 固化到 PDO 模板 | 便于统一状态机处理 |

### 4.4 正极限、负极限与回零设置技巧

#### 正极限设置技巧

1. 先确认机械正方向，再定义 `+EL`，不要只按电机转向猜。
2. 正极限开关应放在“正常工作区外侧，但早于机械硬碰撞点”。
3. 软正限位应比硬件正极限更早触发，并留出减速距离。
4. 正极限触发后，软件层应只允许 `反方向 Jog` 或专用 `Limit Release`。
5. EtherCAT 项目里要同时核对 `I/O 采样值`、`PDO 状态` 和 `HMI 显示逻辑`，避免电平取反。

#### 负极限设置技巧

1. 负极限逻辑与正极限对称，但不要默认参数也完全镜像。
2. 负方向若靠近回零区，要特别确认“回零路径不会先撞负限位”。
3. 双工位或龙门机构里，负极限往往更容易被治具或拖链干涉，现场要单独留安全余量。

#### 回零设置技巧

1. 初学项目优先用“`先找 ORG，再找边沿，再找 Index，最后加 Offset`”的稳定流程。
2. 把回零拆成 `粗找` 和 `精找` 两段速度，粗找快一点，精找慢一点。
3. 原点开关不要贴着正负限位放置，避免回零时没有退让空间。
4. `homeOffset` 不要设为 0 作为默认值，最好让零点离开关触发区几毫米或几度。
5. 回零后立即记录“命令位置、实际位置、原点输入状态、限位状态”，方便复盘。
6. 如果使用绝对值编码器，也建议保留“人工重新回零”路径，便于维护和校准。

### 4.5 热复位与冷复位策略

推荐把复位动作按层次设计，而不是一键全重启：

| 级别 | 典型动作 | 适用情况 |
| --- | --- | --- |
| 一级：报警复位 | 清驱动器故障、清轴错误位 | 普通跟随误差、互锁解除后恢复 |
| 二级：热复位 | 重新初始化轴、重新下发模式、主站重连但不断主电 | 软件状态错乱、PDO 状态残留、单轴偶发异常 |
| 三级：冷复位 | 驱动器重新上电、控制器重启、必要时整机掉电重启 | 驱动器锁死、从站上不了 OP、I/O 电平异常残留 |

建议顺序：

1. 先做报警复位
2. 再尝试热复位
3. 仍不恢复时，再做冷复位
4. 冷复位后如果是增量系统，通常要重新回零

## 5. 典型初始化流程

建议把启动流程做成明确的状态机，而不是散落在多个按钮事件里。

### 启动顺序

1. 加载配置文件
2. 初始化日志系统
3. 连接控制卡或主站
4. 扫描轴、驱动器、I/O
5. 下发必要参数
6. 检查急停、门禁、气压、伺服电源等前置条件
7. 清报警
8. 伺服使能
9. 回零
10. 进入 Ready 状态

### 推荐状态机

```text
PowerOn
  -> Initializing
  -> HardwareConnected
  -> SafetyChecked
  -> ServoReady
  -> Homed
  -> Ready
  -> Auto / Manual
  -> Fault
```

任何一步失败，都应该能回到 `Fault` 并记录错误原因。

## 6. 一套通用的运动开发顺序

### 第 1 步：只打通“读状态”

先不要急着让电机动。先确认以下数据都能正确读取：

- 驱动器在线状态
- 报警状态
- 使能状态
- 命令位置
- 实际位置
- 限位、原点、急停输入

只要状态读不准，后面的动作都不可信。

### 第 2 步：只做“单轴控制”

从最小功能开始：

1. 报警复位
2. 使能
3. 点动
4. 停止
5. 回零
6. 绝对定位
7. 相对定位

### 第 3 步：再做“多轴协同”

等单轴稳定后，再做：

- 双轴直线插补
- 圆弧插补
- 同步跟随
- 龙门
- 飞拍
- 轨迹缓存

### 第 4 步：最后再接业务

业务层常见逻辑包括：

- 配方参数
- 视觉定位
- 扫码
- MES 上传
- 工艺节拍
- 治具状态联锁

这些都应该建立在稳定的运动抽象层之上。

## 7. 伪代码示例

### 7.1 单轴启动与回零

```csharp
public async Task PrepareAxisAsync(IAxis axis, CancellationToken ct)
{
    await axis.ResetAlarmAsync(ct);
    await axis.ServoOnAsync(ct);
    await axis.HomeAsync(ct);
}
```

### 7.2 运动前的安全检查

```csharp
public void EnsureSafeToMove(MachineState state, AxisCommand command)
{
    if (state.EmergencyStopActive)
        throw new InvalidOperationException("EMG active.");

    if (!state.SafetyDoorClosed)
        throw new InvalidOperationException("Safety door open.");

    if (!state.AirPressureReady)
        throw new InvalidOperationException("Air pressure not ready.");

    if (command.TargetPosition < state.SoftLimitMin || command.TargetPosition > state.SoftLimitMax)
        throw new InvalidOperationException("Target exceeds soft limit.");
}
```

### 7.3 标准单轴动作流程

```csharp
public async Task MoveToPickPositionAsync(IAxis axis, double target, CancellationToken ct)
{
    EnsureSafeToMove(_machineState, new AxisCommand(target));

    var profile = new MotionProfile(
        velocity: 200.0,
        acceleration: 800.0,
        deceleration: 800.0);

    await axis.MoveAbsoluteAsync(target, profile, ct);
    await _motionWaiter.WaitInPositionAsync(axis, timeoutMs: 5000, ct);
}
```

### 7.4 C/C++ 风格调用顺序示意

```cpp
connect_controller();
load_axis_parameters(axis_id);
reset_alarm(axis_id);
servo_on(axis_id);
home_axis(axis_id);
move_absolute(axis_id, target_pos, velocity, acc, dec);
wait_axis_in_position(axis_id, timeout_ms);
```

注意：

- 真正函数名会因厂商 SDK 不同而不同
- 但工程顺序大体相同

## 8. EtherCAT 项目要特别注意什么

### 8.1 不要只看“轴在线”

EtherCAT 项目最常见误区是：

- 从站在线了
- OP 了
- 但轴仍然不动

这时通常要检查：

- `CiA 402` 模式是否正确
- `Controlword` 和 `Statusword` 是否走完使能状态机
- `PDO` 是否映射了目标位置、实际位置、控制字、状态字
- 驱动器是否还被硬件互锁、STO、报警、限位拦住

### 8.2 周期线程与 UI 线程要分开

在 Windows IPC 上做 PC-based 控制时：

- UI 刷新不要阻塞运动线程
- 日志写盘不要阻塞总线处理线程
- 视觉处理不要直接卡住轴状态轮询

至少要做到：

- 运动控制线程独立
- UI 只读快照数据
- 异步队列处理日志和历史记录

### 8.3 Windows 不是硬实时系统

要接受一个工程事实：

- Windows 上位机适合做强功能、强集成的控制平台
- 但它不是裸核实时控制器

所以你需要：

- 控制好后台软件和杀毒影响
- 评估网卡、驱动版本和电源管理
- 在真正提高难度前，先把最小周期跑稳

### 8.4 EtherCAT 项目里的回零流程推荐

对多数基于 `CiA 402` 的 EtherCAT 伺服轴，可以把回零理解成 3 层：

1. 总线层：主站与从站进入稳定的 `OP` 状态
2. 驱动层：驱动器进入正确的 `Operation Mode`，常见为 `HM` 或由上位机做位置控制回零
3. 应用层：回零完成后写入统一零点、偏移和状态标志

工程上建议：

- 不要在从站还没稳定进入 `OP` 时就急着发回零命令
- 不要让 UI 直接跳过“回零完成”标志改坐标
- 回零完成后把 `Homed` 状态写入应用状态机，而不是只看驱动器单个标志位

### 8.5 什么时候用热复位，什么时候用冷复位

更适合热复位的情况：

- 软件状态机卡死
- 单轴偶发报警清不干净
- 主站已在线，但 PDO 状态需要重新初始化

更适合冷复位的情况：

- EtherCAT 从站反复上不了 `OP`
- 驱动器面板报警在热复位后仍不消失
- STO、急停、安全继电器恢复后，驱动器状态仍异常
- I/O 电平看起来正常，但驱动器内部状态明显不一致

## 9. 报警与日志设计

现场最能体现工程质量的，往往不是“页面好不好看”，而是报警和日志是否能帮你快速定位问题。

建议至少记录：

- 时间戳
- 轴名
- 动作类型
- 指令位置
- 实际位置
- 速度
- 当前模式
- 报警码
- 触发来源

### 报警对象建议

```csharp
public sealed record AlarmRecord(
    DateTime Timestamp,
    string Code,
    string Source,
    string Message,
    string Severity,
    string? AxisName);
```

### 日志分类建议

- `System`：启动、退出、配置加载
- `Motion`：回零、定位、停止、到位
- `IO`：输入边沿、输出切换
- `Alarm`：故障、复位、保护动作
- `Recipe`：配方切换、参数修改

## 10. 调试阶段的开发顺序建议

如果你们团队正在做第一台样机，推荐按下面顺序推进：

1. 先只接一轴，打通完整启动链路
2. 再补原点与限位
3. 再补报警复位和异常恢复
4. 再补第二轴和插补
5. 再接 I/O 联锁
6. 再接视觉和工艺流程
7. 最后再做性能优化和 UI 完善

这个顺序看似慢，实际上是最快的。

## 11. 常见架构误区

### 误区 1：把底层卡 API 直接暴露给页面

结果：

- 页面代码会越来越重
- 无法做自动流程复用
- 切换卡型成本极高

### 误区 2：没有状态机，全靠布尔变量

结果：

- “能不能动”说不清
- 出错后恢复路径不明确
- 现场 bug 很难复现

### 误区 3：运动、I/O、报警、业务流程都在一个线程

结果：

- UI 卡顿
- 轮询抖动
- 现场偶发问题难以稳定复现

### 误区 4：没有配置化

结果：

- 改一台机型就得改代码
- 参数来源混乱
- 现场备份和恢复困难

## 12. 一套简洁但够用的开发检查表

每次新机型导入前，确认以下问题都有答案：

- 轴的工程单位是什么，如何换算
- 回零方式是什么，是否需要 Z 相
- 软限位和硬限位谁先起作用
- 报警复位是否需要时序或延时
- 运动停止分哪几类：减速停、急停、报警停
- 驱动器模式是否固定，是否允许切换
- 断电重启后位置如何恢复
- 是否要支持手动、自动、维修三种模式

## 13. 给开发者的最后建议

- 不要把“能跑起来”当成“可以交付”。
- 真正可交付的 PCBase 软件，一定有抽象层、状态机、日志、联锁和排障手段。
- 把第一次样机调试中踩过的坑写回文档，比任何培训都更有价值。
